# Projetly — Zapier Integration

The **Projetly** Zapier integration connects Projetly — an AI-powered platform for onboarding, project management, workflow automation, and customer collaboration in digital sales rooms — to the thousands of apps available on Zapier.

It is built on `zapier-platform-core` v19.0.0, runs on Node.js ≥ 22, and authenticates against Projetly using OAuth2 (HMAC-SHA256 signed authorize URL with auto-refresh).

---

## Prerequisites

- **Node.js** ≥ 22 and **npm** ≥ 5.6.0
- A **Zapier developer account** — https://developer.zapier.com
- **Zapier Platform CLI** installed globally:
  ```bash
  npm install -g zapier-platform-cli
  ```
- Access to **Projetly OAuth credentials** (`CLIENT_ID`, `CLIENT_SECRET`) and the Projetly API base URL.

---

## Getting Started (local setup)

```bash
git clone <repo-url>
cd zapier-app

npm install

cp .env.template .env
# fill in the values in .env (see "Environment Variables" below)

zapier-platform login        # authenticate the CLI with your Zapier account
zapier-platform link         # link this directory to an existing Zapier app
                             # (creates / updates .zapierapprc)

zapier-platform test         # run the test suite to verify the setup
```

After `zapier-platform link`, a `.zapierapprc` file is created locally containing the linked app's ID. Do **not** commit it — it is already in `.gitignore` / `.zapierignore`.

## Project Structure

| Path | Role |
|---|---|
| `index.js` | Entry point. Wires up authentication, middleware, triggers, searches, creates, searchOrCreates, flags, and exports the Zapier app definition. |
| `authentication.js` | OAuth2 config: builds the HMAC-SHA256 signed authorize URL, exchanges/refreshes tokens, and labels the connection as `Projetly ({{organisation}})`. |
| `triggers/` | **Zapier triggers** — events that start a Zap. Two kinds live here: event-based REST hooks (`project_created`, `company_created`, `contact_updated`, `deal_created`, `task_updated`, …) and hidden polling triggers (`get_users`, `get_pipeline`, `get_company`, …) used as dynamic-dropdown data sources. |
| `searches/` | **"Find X" operations** — look up an existing record by ID, name, or other filter. Files: `find_company.js`, `find_contact.js`, `find_deal.js`, `find_project.js`, `find_task.js`. |
| `creates/` | **"Create X" actions** — write new records to Projetly. Files: `create_company.js`, `create_contact.js`, `create_deal.js`, `create_project.js`, `create_task.js`. |
| `test/` | Jest test suites that mirror the source folders: `test/triggers/`, `test/searches/`, `test/creates/`. Run with `npm test`. |
| `field_map.json` | Field metadata shared across triggers / creates (input fields, dropdown definitions, etc.). |
| `.env.template` | Template for the local `.env` file — lists every required environment variable with empty values. |
| `.zapierapprc` | Generated by `zapier-platform link`. Holds the linked Zapier app ID. **Never commit.** |
| `.zapierignore` | Files excluded when the CLI bundles and uploads the integration (e.g. tests, README, env files). |

## Environment Variables

Local development reads from `.env`. The deployed integration reads from a per-version env store on Zapier's side (see "Managing Env Vars on the Deployed Integration").

Required keys (from `.env.template`):

| Variable | Purpose |
|---|---|
| `AUTH_URL` | OAuth authorize / token exchange base URL |
| `REFRESH_TOKEN_URL` | Endpoint used by `refreshAccessToken` |
| `CLIENT_ID` | OAuth app client ID |
| `CLIENT_SECRET` | OAuth app client secret (also used as the HMAC-SHA256 signing key) |
| `MARKETPLACE_URL` | Projetly API base URL — used by every `z.request(...)` call |
| `ZAPIER_DEPLOY_KEY` | Deploy key for non-interactive CLI use (CI / scripts). See "Deploy Keys". |

## Versioning & Deployment

**Current Version:** 1.0.1
*Note: Always check the current version in `package.json`, and update the current version here in `README.md` whenever it changes.*

### Where to change the version

Bump the `version` field in `package.json` using semver:

```json
{
  "name": "projetly",
  "version": "1.0.2"
}
```

Zapier reads this on `zapier-platform push` and treats it as the version identifier for the upload. **Each push must use a new version** — you cannot push over an existing one.

### Push a new version

```bash
zapier-platform validate --skip-build     # check schema / structure
zapier-platform test                      # run Jest tests
zapier-platform push                      # upload as a new version
```

`zapier-platform push` and `zapier-platform promote` are **user-only actions** — they deploy to Zapier and affect live users. Confirm before running.

### Promote a version (make it live for new users)

```bash
zapier-platform versions                          # list all deployed versions
zapier-platform promote <version>                 # e.g. 1.0.2
zapier-platform migrate <fromVersion> <toVersion> # move existing users forward
```

### Other useful CLI commands

```bash
zapier-platform logs --type http --detailed       # recent production HTTP logs
zapier-platform invoke trigger <key>              # invoke a single trigger interactively
zapier-platform history                           # change history for the integration
zapier-platform users:list                        # list users invited to the integration
zapier-platform jobs                              # ongoing migration / promotion jobs
zapier-platform describe                          # describe the linked integration
```

## Deploy Keys

A deploy key is what the CLI uses to authenticate against Zapier. On a normal dev machine, running `zapier-platform login` writes it into `~/.zapierrc` for you. For CI / scripted deploys (no interactive login), you supply the key via an environment variable instead.

**Generate / manage deploy keys** in the Zapier developer dashboard:

1. Sign in at https://developer.zapier.com
2. Open your integration → **Manage → Deploy Keys**
3. Create a new key and copy the value (shown only once).

**Use the deploy key**

Interactive (one-time setup on a dev machine):

```bash
zapier-platform login          # prompts for the key and writes ~/.zapierrc
zapier-platform logout         # clears it
```

Non-interactive (CI / scripts) — export `ZAPIER_DEPLOY_KEY` in the shell before running CLI commands:

```bash
# macOS / Linux
export ZAPIER_DEPLOY_KEY=<your-key>

# Windows PowerShell
$env:ZAPIER_DEPLOY_KEY = "<your-key>"

zapier-platform push
```

Reference: https://docs.zapier.com/integrations/build-cli/overview

## Managing Env Vars on the Deployed Integration

Your local `.env` is only used for `zapier-platform test` and `zapier-platform invoke`. The **deployed** integration has its own per-version environment store, managed via the `zapier-platform env` topic, which has three subcommands:

```bash
# Get all env vars set on a deployed version
zapier-platform env:get <version>

# Set one or more env vars (KEY=VALUE pairs, space-separated)
zapier-platform env:set <version> KEY1=VALUE1 KEY2=VALUE2

# Unset one or more env vars (space-separated keys)
zapier-platform env:unset <version> KEY1 KEY2
```

Examples:

```bash
zapier-platform env:get 1.0.1
zapier-platform env:set 1.0.1 CLIENT_ID=abc123 MARKETPLACE_URL=https://api.projetly.com
zapier-platform env:unset 1.0.1 CLIENT_ID
```

Notes:
- Keys are case-insensitive.
- Values are copied forward when a new version is created, but each `env:set` / `env:unset` only affects the version you name.
- Pass `-f` / `--force` to modify a production version (use with care).
